# 0.2.30 版本更新

## 新增功能

1. 模型目录集成监控状态与快捷配置
    - 模型卡片支持展示监控状态、7 天可用率和响应延迟。
    - 管理员可从模型目录一键为模型创建监控配置，并自动带入模型名称和提供商信息。
    - 按分组视图新增分组筛选器，便于快速定位特定分组下的模型。

2. 模块中心与渐进式模块能力扩展
    - 公告、邀请记录、返现记录、提取记录、兑换码、优惠码、渠道定价、IP 管理等能力支持独立启用和禁用。
    - 邮件验证、登录条款等能力纳入模块中心，支持统一开关和配置跳转。
    - 新增渐进式组件规范，为后续模块接入提供统一结构。

3. 登录与认证管理页面
    - 新增登录与认证页面，集中管理 OIDC、GitHub、Google、微信、LinuxDO、钉钉等登录配置。

## 修复问题

1. 修复兑换码模块在模块市场中无法稳定启动的问题。
    - 修复 public settings 强制刷新时旧响应覆盖新状态的竞态。
    - 模块市场启用内置模块后会立即同步本地状态，避免开启后又被旧配置打回关闭。

2. 修复渐进式模块开启后界面消失的问题。
    - 前端启用状态、部署模式与路由注册统一同步，避免菜单已显示但路由未注册，或路由已注册后又被旧配置移除。
    - 对直接访问已关闭模块路径的情况增加稳定回退，避免落入异常空白页面。

3. 修复模型目录页面可能出现空白的问题。

4. 修复自定义 Provider 模型列表回退逻辑。
    - 自定义 Provider 的 model_mapping 键会继续作为可见模型来源，避免模型列表为空。

## 优化调整

1. 全面收拢渐进式模块结构
    - 前端模块定义集中到 `utils/progressiveFeatures.ts`，渐进式路由集中到 `router/progressiveRoutes.ts`。
    - 后端模块定义集中到 `service/progressive_features.go`，API 入口统一通过 `RequireProgressiveFeature` 中间件守卫。
    - 新模块接入时只需在集中注册表中声明开关、路由和后端能力，减少散落在各页面和 handler 中的硬编码判断。

2. 前后端启停语义统一
    - 模块未开启时，前端不注册对应路由、不显示菜单、不下载 lazy chunk。
    - 模块未开启时，后端对应 API 在进入业务 handler 前返回 404，不启动相关后台任务。
    - 模块开启后，前端路由和菜单立即恢复，后端 API 与后台任务同步激活。

3. 后台任务生命周期优化
    - 渠道监控 runner 支持按开关暂停和恢复，关闭模块时停止 ticker 和在途任务上下文。
    - 支付订单过期扫描按支付模块开关启动和停止。
    - 风控内容审核 worker 和清理任务按风控模块开关启动和停止。

4. 侧边栏与系统设置体验优化
    - 管理后台菜单按功能模块分组，用户端菜单按使用场景分组。
    - 分组支持折叠展开，收起状态下点击图标可直接进入对应功能。
    - 设置页面 Tab 样式调整为底部边框风格。
    - 侧边栏版本号显示更简洁，Preview 版本通过颜色区分。

5. 运维与管理效率优化
    - 升级弹窗新增立即重启按钮。
    - 账户管理、分组管理、渠道管理等表格支持更多列排序。
    - 模型目录来源数量徽章样式与其他页面保持一致。

## 兼容性 / 破坏性变更

1. 渐进式模块关闭后的后端行为更严格
    - 影响范围：已关闭模块对应的前端页面、用户 API、管理 API、支付公共查询与支付 webhook。
    - 处理方式：需要继续接收某类模块请求时，请先在模块中心或系统设置中启用对应模块；支付业务仍在使用时请保持支付模块开启。

2. 个人模式会结构性移除分发模式专属能力
    - 影响范围：公告、兑换码、优惠码、订阅、风控等分发模式专属入口。
    - 处理方式：需要使用这些能力时，将部署模式切换为分发模式，并确认对应模块开关已开启。

## 升级说明

1. 升级到 0.2.30 后建议进入模块中心检查模块开关状态。
    - 已有配置会继续保留，但关闭的模块将不再显示前端入口，也不会激活后端 API 或后台任务。
    - 如果需要使用兑换码、公告、优惠码、支付、风控、渠道监控等功能，请确认对应模块处于开启状态。

2. 本次正式版不要求新增环境变量或数据库手动迁移。
    - 版本号已更新为 `0.2.30`。
    - 建议升级后执行一次服务重启，确保后台任务生命周期与模块开关状态完全同步。
