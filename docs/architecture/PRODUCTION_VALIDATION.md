# Production Validation Record

This record describes the validation performed on the phase-three progressive
feature implementation. It distinguishes checks that were actually executed in
the offline build sandbox from checks that still require the connected GitHub CI
environment.

## Persisted baseline

The production tree is based on the persisted phase-three frontend/module
runtime milestone. Final lifecycle consistency and documentation changes are
recorded in two local commits:

```text
b7a9d9a feat: finalize progressive feature runtime and module loading
2468fc6 fix: make progressive feature startup atomic
```

The repository inventory is generated by `tools/codebase_inventory.py` and is
checked into `docs/architecture/CODEBASE_INVENTORY.tsv`.

## Gates executed in the sandbox

- repository inventory regeneration and SHA-256 round-trip check;
- Go parser validation for 1,782 Go files;
- TypeScript parser validation for 687 TS/JS/Vue script units;
- resolution check for 2,054 local frontend imports;
- JSON and non-template YAML parsing;
- `bash -n` for checked-in shell scripts;
- merge-conflict marker and patch whitespace checks;
- high-signal committed-key marker scan;
- ZIP integrity, required-file, no-`.git`, extraction and inventory checks;
- isolated `go test -race` against the exact `FeatureRuntimeManager` core,
  covering atomic multi-component startup rollback and dynamic pause/resume.

The isolated lifecycle harness uses only the Go standard library because the
sandbox has no module cache or network access. It copies the current runtime
manager implementation and supplies minimal catalog/configuration stubs; it is
not a substitute for the repository test suite, but it validates the new state
machine itself under the race detector.

## Required connected CI gates

The project requires Go 1.26.4. The sandbox provides Go 1.23.2, has no Go module
cache, no pnpm installation and no frontend `node_modules`. The following gates
therefore remain mandatory before deployment and must run in GitHub Actions or
an equivalent connected build environment:

```bash
# Backend
cd backend
go version                         # must report go1.26.4
make test-unit
make test-integration
golangci-lint run --timeout=30m ./...
go test -race -tags=unit ./...
CGO_ENABLED=0 go build -trimpath ./cmd/server

# Frontend
cd frontend
pnpm install --frozen-lockfile
pnpm run lint:check
pnpm run typecheck
pnpm run test:run
pnpm run build
```

PostgreSQL and Redis integration services must use the same versions and
configuration as `.github/workflows/backend-ci.yml`. A release must not be
promoted when any mandatory CI or security-scan job is red.

## Deployment checks

1. Select `features.profile` deliberately (`minimal`, `standard` or `full`).
2. Confirm every `boot` feature with `requiresRestart=true` has been applied by
   restarting the process before traffic is admitted.
3. Verify `/api/v1/settings/features` exposes no runtime errors or paths.
4. Verify `/api/v1/admin/features/runtime` is accessible only to administrators.
5. Exercise module enable, disable and version upgrade; confirm old-version
   assets return 404 and module routes disappear when disabled.
6. Confirm graceful shutdown drains the feature runtime before Redis and
   PostgreSQL are closed.
7. Monitor pending pricing failures, billing queue fallback metrics and module
   runtime errors during the first rollout.

## Rust decision

No Rust component was introduced. The measured work in this phase is dominated
by database, Redis, network and filesystem latency. Batching, immutable snapshots,
single-flight refresh and concurrent independent probes reduce that cost without
adding an FFI or sidecar release surface. Rust remains appropriate only after a
CPU profile identifies a stable isolated compute kernel.
