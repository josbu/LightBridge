## 后端修复（账号重复迁移与删除账号重现）

### 问题根因
**OpenAI 自动迁移**（`modulemigration.RunAutoOpenAIModuleMigration`）在每次版本升级/重启时会运行，原实现存在两个致命缺陷：

1. **INSERT 模式产生重复账号**：旧逻辑为 `platform='openai'` 账号创建一个全新的 `platform='module'` 副本，但**从不清理源 `openai` 行**。结果：
   - 原 `openai` 账号依然存在 → 成为**永久重复账号**。
   - `hasLegacyOpenAIAccounts()` 查询永远返回 true → 每次启动都重新执行迁移 → 每次都再插入一份 `module` 副本。

2. **已删除账号被重新添加**：虽然有 `skipDeletedOpenAIAccount` 守卫，但由于迁移标识存储在 `extra->'module_migration'` 中，当用户删除了已迁移的 `platform='module'` 账号后，源 `platform='openai'` 行仍然存在且未被标记。下次迁移时：
   - 查不到已迁移的 `module` 账号 → 视为"未迁移"。
   - 重新 INSERT 一份 `module` 账号 → **已删除账号"复活"**。

### 解决方案
**同库迁移使用就地转换（in-place conversion）**：当 source 与 target 指向同一数据库时（启动自动迁移必然如此），不再插入新行，而是：

1. **UPDATE 源账号的 `platform` 字段**：`UPDATE accounts SET platform='module', ... WHERE id=<legacy_id> AND platform='openai' AND deleted_at IS NULL`。
2. **保持 account.id 不变**：API Keys、分组关联、用量记录全部指向同一 id，无需重新映射。
3. **0 行受影响时静默跳过**：若源行已删除/已转换，UPDATE 无匹配行 → 不插入副本 → **不会重复，不会复活**。

### 核心改动

#### `backend/internal/modulemigration/migrator.go`
- **新增 `Options.SameDatabase` 标志**：标识 source/target 为同一数据库。
- **`upsertOpenAIAccount` 分支逻辑**：
  - `SameDatabase=true` 且 `LegacyID` 为数字 → 执行 `UPDATE accounts WHERE id=<legacyID> AND platform='openai'`。
  - 否则走原 INSERT 逻辑（跨库迁移场景）。
- **新增 `numericLegacyID` 辅助函数**：安全解析 legacy_id 为 int64。

#### `backend/internal/modulemigration/auto.go`
- `RunAutoOpenAIModuleMigration` 中显式设置 `SameDatabase: true`。

#### `backend/internal/modulemigration/migrator_test.go`
- **`TestUpsertOpenAIAccountConvertsInPlaceForSameDatabase`**：验证同库迁移执行 UPDATE 而非 INSERT。
- **`TestUpsertOpenAIAccountSkipsNonNumericLegacyIDForSameDatabase`**：验证非数字 legacy_id 时静默跳过（避免误操作）。

### 附加清理
- 删除 `backend/internal/server/routes/gateway.go` 中未使用的 `routeCustomGroupGemini` 函数（遗留代码）。

## 升级影响评估

### 行为变更
- **首次应用 v0.1.7 时**：现有 `platform='openai'` 账号会被就地转换为 `platform='module'`（保持 id 不变）。
- **后续重启时**：`hasLegacyOpenAIAccounts()` 返回 false → 自动迁移跳过，**不再每次启动都重跑**。

### 兼容性
- **向前兼容**：已完成迁移的 v0.1.5/v0.1.6 实例升级到 v0.1.7 时，若已存在 `platform='module'` 副本，迁移会检测到并跳过（UPDATE 0 行 → 无操作）。
- **数据完整性**：就地转换保留 account.id，所有外键关系（API Keys、Groups、UsageLogs）自动延续。

### 回滚计划
若需回滚到 v0.1.6，可手动执行：
```sql
UPDATE accounts 
SET platform = 'openai' 
WHERE platform = 'module' 
  AND extra->>'provider_id' = 'openai';
```
